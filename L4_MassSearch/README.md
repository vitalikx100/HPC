# Лабораторная работа №4

**Массовый поиск подстрок с использованием CUDA**

**Задача**: произвести поиск множества подстрок различной длины в буфере данных. Поиск производится на полном восьмибитном алфавите

**Язык**: C++ или **Python**

**Входные данные**: строки, состоящие из случайных последовательностей байт, параметры входных данных (длина буфера для поиска подстрок, количество подстрок, минимальная и максимальная длина подстроки, вариант поиска - поиск всех позиций подстроки
в строке)

**Выходные данные**: время работы алгоритма с использованием GPU, время работы алгоритма без использования GPU, отчет о совпадении/отличии результатов работы алгоритма с и без использования GPU, результаты поиска

# Реализация
**CPU**

Функция **mass_search_cpu** реализует последовательный поиск подстрок в буфере на **CPU**: сначала создается матрица **R** (размер: количество паттернов * длина буфера), где каждая строка инициализируется длиной соответствующей подстроки. 

Далее с помощью функции **build_pair_tables** создается структура данных - для каждого из возможных байтов создается список пар, в которых указывается, в каких подстроках и на каких позициях встречается определененный символ. 

Затем выполняется обход буфера: для каждого символа извлекается список пар, для каждой пары вычисляется возможная позиция начала подстроки **col = i - off**. Если позиция начала подстроки находится в допустимых пределах (0 <= col < buf_len), то счетчик в матрице **R** по адресу **R[pid, col]** (pid - индекс подстроки) уменьшается на 1. 

В завершении функция **collect_results** определеяет в матрице R позиции, где значения = 0 (0 - полное совпадение подстроки).

**GPU**

Функция **mass_search_gpu** реализаует **параллельный** поиск подстрок на **GPU** с использованием **CUDA**: исходный буфер, матрица R и таблицы пар копируются в видеопамять при помощи **cuda.memcpy_htod**. **Размер блока** = 256 потоков, размер сетки рассчитывается по формуле **(buf_len + block_size - 1) // block_size** для покрытия буфера.

Запускается CUDA ядро, где каждый поток обрабатывает **один символ** буфера. Поток получает символ, по нему определяет список пар и для каждой пары вычисляется возможная позиция начала подстроки **col = i - off**, после чего **атомарно уменьшает** соответствующий счетчик в матрице **R** на 1 при помощи **atomicSub**.

После завершения результат копируется в оперативную память с помощью **cuda.memcpy_dtoh** и выполняется сбор найденных позиций подстрок (как на CPU).

Полученные результаты сравниваются при помощи функции **compare_results** на корректность.
# Параметры входных данных программы
**--buf_len** - длина буфера данных

**--n_patterns** - количество подстрок

**--min_len** - минимальная длина подстроки

**--max_len** - максимальная длина подстроки

**--seed** - повторяемость эксперимента

# Полученные результаты эксперимента
**Размер буфера данных**

Cначала проводится **анализ ускорения в зависимости от размера буфера данных (buf_len)** при фиксированном количестве паттернов = 50, минимальной длины подстроки = 3 и максимальной длины подстроки = 10. 

Рассматриваются следующие **размеры буфера данных**: 500, 1,000, 10,000, 50,000, 100,000.

Запускается при помощи команды **python L4_masssearch_6132_StaryginVA.py --buf_len 500 --n_patterns 50 --min_len 3 --max_len 10 --seed 123** с дальнейшим варьированием **--buf_len**

| Размер буфера данных | Время CPU, мс | Время GPU, мс | Ускорение, раз | Совпадение результатов |

| 500                  | 0,94          | 0,77          | 0,82           | да                     |
| 1,000                | 1,30          | 1,01          | 1,29           | да                     |
| 10,000               | 10,66         | 2,20          | 4,84           | да                     |
| 50,000               | 50,36         | 7,03          | 7,16           | да                     |

Время выполнения на CPU увеличивается пропорционально размеру буфера данных, показывая линейную зависимость. При увеличении размерности с 500 до 100,000 время возрастает с 0,94 мс до 96,79 мс.

Время выполнения на GPU также растет с увеличением буфера данных, но существенно медленнее. При малом размере накладные расходы на копирование данных и запуск ядра сопоставимы с вычислительной нагрузкой. 
Но с дальнейшим увеличением размерности буфера вычисления на GPU показывают эффективную параллельную обработку.

При малом размере буфера (500 элементов) наблюдается замедление параллельного алгоритма в связи с накладными расходами. 
При дальнейшем увеличении буфера данных до 100,000 ускорение возрастает до 7,54 раз, что показывает преимущество параллельного алгоритма для обработки большого объема данных.
